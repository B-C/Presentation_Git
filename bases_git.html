Bases de Git 

 

Essayons de voir ce que tout se passe dans un projet. Au départ, nous supposons que notre projet est déjà présent dans certains externes « dépôt « . Pour télécharger ce code, nous utilisons la commande suivante

git clone <URL>

par exemple:-

  1. git clone elecinf381@hg.comelec.enst.fr:2011/PrènomNom

Nous avons une copie du référentiel externe dans notre système local. Maintenant, si nous voulons ajouter un nouveau fichier à notre projet, nous créer le fichier, eg. EthernetDriver.c, mais Git ne sait rien sûr cette fichier . Donc, l'état actuel du fichier est "Untracked".

  1. Emacs EthernetDriver.c (Écrire le code et sauvegardez le fichier)

                Le répertoire dans lequel nous travaillons est appelé « répertoire de travail » ou « Working Directory «. Nous faisons codage dans ce répertoire.

Pour obtenir le statut d'aucun fichier, nous faisons git status <filename> ou juste git status

  1. git status Etherenet.c

# On branch master

# Untracked files:

#   (use "git add <file>..." to include in what will be committed)

#

#             EthernetDriver.c

nothing added to commit but untracked files present (use "git add" to track)

Si nous faisons git status sur EthernetDriver.c après sa création nous voir que Git rapports à nous qu'il est « Untracked « .

D'ajouter ce fichier dans dépôt local, nous utilisons la commande  git add <filename>

  1. git add EthernetDriver.c

Après cela, git commencera suivi ce dossier et il change le statut du fichier de « Untracked« à « Tracked «. Un fichier qui peut-être déjà suivis par git a le statut "Intact" sauf si vous faites certains changements. Le statut du fichier devient alors ''modification".

Après avoir fait les changements, idéalement, nous devrions tester nos modifications et une fois nous sommes satisfaits nous puissions procéder de stocker ces changements en permanence à notre dépôt local. Ceci est appelé "committed" un fichier. Dans Git, tous les changements qui sont d'être commis sont d'abord stockées dans une zone appelée "Staging Area". Pour ce faire, nous allons utiliser la commande encore git add.

git add <filename>

  1. git add EthernetDriver.c

Cette commande va prendre les changements présents dans le dossier et va stocker dans le "staging area". L'état du fichier est maintenant « staged « . Si nous modifier le fichier après cela, le changement ne serait pas reflété dans «staged area «  et, partant ne seront pas stockés dans notre dépôt sur le prochain commettre. Pour mettre à jour tout autre changement qui serait arrivé dans le fichier après «staging«,nous utiliser la commande

git add –u

Pour supprimer un fichier d'un dépôt git utilisation

git rm <filename>

Et à supprimer tout fichier de la zone «staging area« , l'utiliser avec l'option

 git rm --cached <filename>

De commettre un fichier nous utilisons

  1. git commit

Seuls les changements présents dans la zone « staging area » obtiendrez commis. Si nous ajoutons le nom de fichier comme

Git commit <Filename>

Alors, nous n’avons pas besoin de "stage" le fichier.

En utilisant l'option -a avec les commettre commande automatiquement "ajouter" changements des fichiers (à savoir tous les fichiers qui sont déjà énumérés) et à automatiquement "rm" des fichiers dans l'index qui ont été retirés du travail arbre, et ensuite d'exécuter la commettre.

 

 

Voir l'ensemble l'histoire «commit« d'un fichier nous pouvons utiliser

git log <filename>

git log EthernetDriver.c

Utilisez –p avec la commande ci-dessus pour afficher le patch aussi

Afin d'afficher les différences entre les différentes versions d'un fichier nous utilisons la commande

git diff <filename>

Cette affiche les différences en ce qui concerne répertoire de travail. De voir les changements présents dans la zone «staging area «  utiliser l'option –cached avec la commande ci-dessus.
